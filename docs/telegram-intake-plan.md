# Telegram Intake Plan

## Purpose

This document defines a safe staged Telegram workflow for shop alerts and pasted ticket intake.

The goal is to support operational visibility first, keep intake simple, and avoid risky automation until the intake path is stable and reviewable.

This plan is advisory-first and should feed the existing observer flow without introducing direct system access.

## Stage 1: Alerts Only

Stage 1 should support one-way alert delivery from the observer to Telegram.

Primary use:

- send shop summary alerts
- send priority item alerts
- send follow-up reminders

Stage 1 should not attempt to parse ticket updates from Telegram yet.

Recommended behavior:

- send concise alert messages
- include ticket reference, advisor, location, priority, and exception summary
- link alert content to the same categories already used by the observer

## Stage 2: Pasted Ticket Intake

Stage 2 should allow a user to paste structured ticket updates into Telegram for observer processing.

Primary use:

- quick manual ticket updates
- end-of-day or mid-day shop status refreshes
- lightweight intake when no direct integration is available

Stage 2 should only accept clearly structured text messages.

The intake path should convert accepted messages into monitored item records that can be written into `current_shop_snapshot.json`.

## Stage 3: Screenshot/Photo Intake Later

Screenshot and photo intake should be deferred until the text intake path is reliable.

Primary future use:

- reading screenshots of ticket summaries
- reading photos of handwritten or printed shop notes
- converting visual input into structured ticket records for review

This stage should require extra review because screenshots and photos are more error-prone than structured text.

Screenshot/photo intake should be treated as draft intake only.

Extracted data must be reviewed before updating `current_shop_snapshot.json`.

Screenshot/photo intake should never overwrite live data automatically.

Parsed visual intake should first be written into an `inputs/intake_drafts/` folder concept for review, not directly into the live snapshot path.

Recommended draft flow:

- receive screenshot or photo in Telegram
- extract structured draft ticket data
- assign a confidence level
- save draft output for review
- ask for confirmation or corrected pasted text before any live snapshot update

Confidence levels should be defined as:

- high confidence: can suggest direct update
- medium confidence: ask confirmation
- low confidence: ask user to paste text instead

## Required Telegram Message Formats

Stage 1 alert messages can be generated by the system.

Stage 2 intake messages should use a structured format such as:

```text
TICKET
item_id: ITEM-201
location: Country Club
advisor_name: Drew
ticket_reference: RO-2401
current_status: estimate_pending
last_update_at: 2026-04-24T14:30:00
has_parts_hold: false
dvi_complete: true
estimate_complete: false
status_mismatch: false
follow_up_needed: true
notes: Waiting for customer approval on estimate.
```

Multiple pasted tickets should use repeated `TICKET` blocks in the same message or separate messages.

Boolean fields should use:

- `true`
- `false`

Timestamp format should use ISO format:

- `YYYY-MM-DDTHH:MM:SS`

## What The Bot Should Accept

The future bot should accept:

- observer alert delivery requests
- single structured ticket blocks
- multiple structured ticket blocks in one message
- clearly labeled correction messages for an existing ticket
- requests to validate whether a pasted ticket format is usable

The bot should accept only the fields currently supported by the observer data structure:

- `item_id`
- `location`
- `advisor_name`
- `ticket_reference`
- `current_status`
- `last_update_at`
- `has_parts_hold`
- `dvi_complete`
- `estimate_complete`
- `status_mismatch`
- `follow_up_needed`
- `notes`

## What The Bot Should Reject Or Ask Clarification On

The future bot should reject or request clarification for:

- missing required fields
- unsupported field names
- unclear timestamps
- freeform messages with no ticket structure
- mixed ticket details that cannot be separated cleanly
- requests containing credentials or access tokens
- requests to log into Tekmetric or Autoflow
- requests to change systems directly
- destructive or execution-style commands

The bot should ask for clarification when:

- location is missing or ambiguous
- advisor name does not match expected shop roles
- ticket reference is missing and item identity is unclear
- status value does not match the expected observer vocabulary
- multiple tickets are blended into one block

## How Messages Map Into `current_shop_snapshot.json`

Accepted Telegram intake messages should map one ticket block to one JSON object.

Field mapping should remain direct:

- `item_id` -> `item_id`
- `location` -> `location`
- `advisor_name` -> `advisor_name`
- `ticket_reference` -> `ticket_reference`
- `current_status` -> `current_status`
- `last_update_at` -> `last_update_at`
- `has_parts_hold` -> `has_parts_hold`
- `dvi_complete` -> `dvi_complete`
- `estimate_complete` -> `estimate_complete`
- `status_mismatch` -> `status_mismatch`
- `follow_up_needed` -> `follow_up_needed`
- `notes` -> `notes`

Target write behavior for a later implementation:

- validate message structure first
- normalize fields second
- generate a clean JSON array third
- write the validated snapshot to `inputs/current_shop_snapshot.json`

If validation fails, the bot should not write the snapshot file.

For screenshot/photo intake later:

- parse extracted content into draft JSON first
- save draft output under an `inputs/intake_drafts/` folder concept
- require human review before promoting draft data into `inputs/current_shop_snapshot.json`
- block automatic overwrite of live snapshot data from visual intake alone

## Safety Rules

- No credentials in Telegram.
- No destructive actions.
- No direct Tekmetric or Autoflow login yet.
- No automatic overwrite of live data from screenshot or photo intake.
- No execution of shell commands from Telegram input.
- No direct editing outside the approved snapshot file path.
- No silent acceptance of malformed ticket data.
- No automatic business-system changes.
- No automatic customer communication.

Telegram should be treated as a convenience intake channel, not a trusted system-of-record.

## Next Implementation Steps

1. Define the exact allowed Telegram ticket format and keep it narrow.
2. Add a parser that validates structured ticket blocks without writing files yet.
3. Add a validation report that tells the user what was accepted or rejected.
4. Add controlled writing to `inputs/current_shop_snapshot.json` only after validation passes.
5. Add Stage 1 outbound alerts for observer summaries and priority items.
6. Add test cases for valid tickets, invalid tickets, partial tickets, and mixed multi-ticket messages.
7. Add an `inputs/intake_drafts/` review path for screenshot and photo parsing later.
8. Add confidence scoring for visual intake with high, medium, and low confidence handling.
9. Defer screenshot and photo intake until text intake is stable and reviewable.
